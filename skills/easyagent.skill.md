# EasyAgent 设计范式

## 六条核心原则

### 1. 最小操作次数
用户从说出名字到用上，除必要的询问外，动手次数越少越好。自动化部分由 Agent 通过设计尽可能一次性处理完再找用户。

在需要用户操作时，以最简化用户操作为目标。如设计合理时机时的"可能无需操作点校验"与提供超链接等方式（方式不限），来大幅缩短用户使用时的难度、提升顺滑度。

### 2. 可选项前置，一次性询问
所有选项在开头列清，Agent 一次性问完用户需要哪些，而不是做完一步问一步。

### 3. 自动化灵活构建
根据用户选择自动判断路径。零配置项直接注册，需依赖项自动安装，需登录项最后集中引导。

### 4. 模板优先
有模板时严格按模板执行。没有模板时，完成配置后把流程沉淀为模板。

### 5. 备用方案以用户最小感知量和正确输出为第一顺位
主方案不可用时，备用方案优先考虑用户无感知和输出正确性。用户不需要知道主方案挂了，模型兜底输出正确结果后末尾提一句即可。

### 6. 避免模型执行无意义操作
模型遇到错误时倾向于自行修复（重试命令、改参数等），通常无效且浪费大量时间。明确禁止自行修复，主方案失败直接走备用方案，备用方案仍不行则跳过、如实告知用户。

## 执行流程

```
用户: "帮我装 EasyAgent-XXX"

Agent:
  1. 读 README，找到可选项清单
  2. 一次性问用户需要哪些
  3. 自动执行所有可自动化部分
     - 零配置项直接注册
     - 需依赖项自动安装（npm/pip）
     - 需构建项按模板生成
  4. 调 doctor() 自检确认
  5. 集中告知用户需手动操作的部分 + 给出指引
  6. 用户完成手动操作 → 全部就绪
```

## 产品结构

```
repo/
├── README.md    # 配置入口 + 可选项清单（开头列清）
├── template/    # 模板
├── servers/     # 成品（即用型）
└── skills/      # 可选的 skill 文件，供 Agent IDE 加载
```

## 关键设计决策

- README 的主要读者是 AI Agent，不是人类。开头必须有一段可选项清单。
- 每个成品必须有 doctor() 自检，Agent 靠它确认状态，不靠问用户。
- 备用方案不在代码里硬编码多后端路由，而是写在错误消息里指引模型兜底。
- 模型收到"无可用后端"时，错误消息必须包含：问题原因、替代方案、禁止自行修复、兜底的兜底。
